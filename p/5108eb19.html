【Godot笔记】队列、优先队列与索引优先队列

Giresharu
Godot引擎

发布于：2021年8月15日
次浏览

  Godot中常用的内置数据结构只有两个，一个是 Array ，另一个是 Dictionary 。其中 Array 包含了 pop_back() 和 push_back() 函数，可以直接作为先进后出的栈使用。虽然它也包含了 pop_front() 函数，可供返回并删除索引 [0] 的元素。但官方的文档上却指出，该函数将索引 [0] 的元素抛出后，要将后面的元素遍历向前移动，效率十分低下，不应该直接当做队列使用。在使用 BSF 广度优先算法等情况，我们又十分需要使用先进先出的队列，所以我们必须要自己实现一个队列。

队列 Queue

· 原理

队列的特征

  • 队列是先进先出（FIFO）的，拥有队头和队尾
  • 只能在队头进行删除，即出队
  • 只能在队尾进行插入，即入队

  我们可以利用 Array 来存储队列中的元素，然后通过 front 以及 rear 属性分别来记录队头与队尾在 Array 的索引。这样我们增删数据只需要移动头尾的位置，而不需要伤筋动骨地移动剩余的每个元素。如下面表格所示

索引 0 1 2 3 4
队头队尾 front rear
元素 null 100 406 350 null

  可以观察到，队头指向当前最靠前的元素，每出队一次便会加 1 ，使下个元素成为队头。而队尾则指向当前最靠后的元素的下一个空位，每入队一次便会加 1 ，使下个空位成为队尾。当队列所有元素都被删除时，队头将会和队尾重合。

  但是这样的队列很快就会因为出队入队而用完，前面的空间无法再次使用，全部浪费掉。所以人们发明了一种叫做循环队列的写法：将队列的头尾相连，当队头和队尾越界时，回归到索引 [0] 继续循环。
  然而这样的队列仍然有一个问题，那就是判断队列是否为空的方法失效了：当队满时，队尾将会超过队头一圈，再次和队头重合，与判断为空的条件重复了。
  于是我们必须放弃掉队列的最后一个空位，将还剩下一个空位才满的队列直接当作满，不允许对最后一个空位进行写入。即：在生成队列的时候，多创建一个空位。当队满的定义更改为只剩下一个空位，可以通过验证队尾是否在队头的前一个位置来验证。

队头与队尾的运算

  • 循环队列类似圆环跑道，可以通过取模运算来将越界的值变为界内的值
  • 队尾加 1 时， rear = (rear + 1) % capacity （队头同理）
  • 当 front == rear 时，队头和队尾重叠，说明当前一个元素也没有，队列为空。
  • 当 front == (rear + 1) % capacity 时，队头的前一个位置为队尾，说明队列为满（capacity是包括浪费空间的队列容量）

优先队列 PriorityQueue

· 原理

  优先队列与队列不同，并非先进先出，而是让优先度最高的元素先出。为了不让问题复杂化，在这里我们先只考虑 int 类型的数值大小作为优先度，让数值越小的元素越优先。要实现这样的优先队列，我们需要使用完全二叉树结构来实现。我们将队列视作如图的排列：

亲子节点计算

  • 亲节点: (index - 1) / 2
  • 左子节点： index * 2 + 1
  • 右子节点： index * 2 + 2

  我们只要保证亲节点一定比子节点优先度高，就可以保证索引 [0] 一定是优先度最高的元素。为了保证这点，当我们添加新元素时，要将新元素插在队尾，然后进行上浮操作：

上浮 swim

  • 和亲节点进行比较：如果亲节点的优先度比自己低，则交换位置
  • 重复上一步，直到发现亲节点的优先度更高，或已经没有亲节点时停止

  这样，索引 [0] 的元素一定是优先度最高的。我们出队时，只要认准它即可。

  当我们将索引 [0] 出队时，这个位置会空缺。我们可以将最后一个元素放到顶端填补空位，然后将它下沉：

下沉 sink

  • 和两个子节点比较：当只有一个子节点优先度比自己高时，与之交换位置；当两个子节点优先度都比自己高时，和更高的那个子节点交换位置
  • 重复上一步，直到发现没有比自己优先度更高的子节点，或已经没有子节点时停止

  经过下沉，我们又能将优先度最高的元素放在索引 [0] ，并保证了所有亲子节点的优先度关系正常，不会对后续的上浮下沉造成干扰。

索引优先队列 IndexPriorityQueue

· 原理

  索引优先队列如其名，是从优先队列改造而来。优先队列因为元素一直在改变索引，我们无法高效率地对其元素进行修改。而在如 Dijkstra 算法等情况中，我们又十分需要中途改变优先队列中的元素。
  试想如果我们始终保持 elements 中的元素不移动，另外申明一个 pq 数组来记录元素们各自的索引（即 pq 的元素是 elements 的索引）。在原本需要移动元素的时候，改为移动 pq 中的元素，这样 elements 中的元素位置就不会改变，我们便能轻易的修改他们。如下面表格所示：

索引 0 1 2 3 4 5 6
pq 4 3 5 1 2 6 0
elements 999 45 78 16 0 44 63

  不过这样还不够，当我们修改元素时，已知的是 elements 的索引，即 pq 的元素，而非 pq 的索引。所以我们无法高效地找到对应的索引对 pq 进行上浮下沉。
  所以我们需要再加一个 qp 数组，将 pq 的元素作为索引，并将 pq 的索引作为元素。在上浮下沉的时候，同时对 qp 进行修改。这样我们便能快速地找到需要上浮下沉的 pq 了。

索引 0 1 2 3 4 5 6
pq 4 3 5 1 2 6 0
qp 6 3 4 1 0 2 5
elements 999 45 78 16 0 44 63
  • pq 指的是 PriorityQueue
  • qp 则指代其与 pq 是调换关系，能够互查元素的索引
  • qp[pq 的元素] = pq 的索引
  • pq[qp 的元素] = qp 的索引
